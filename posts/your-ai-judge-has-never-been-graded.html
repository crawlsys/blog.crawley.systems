Your AI Judge Has Never Been Graded

I built an automated vision-model gate for archive fidelity. It looked rigorous. It was an uncalibrated model opinion with a progress bar. The fix was a dumb HTML grid and twenty minutes of actually looking.

I run a web-archiving service. It captures a page, and the recurring question — the only question that actually matters — is: did the capture come out right? Is the snapshot we stored a faithful rendering of the thing we archived, or is it a broken page with missing images, stripped CSS, and a half-loaded body?

You cannot eyeball thousands of these. So I did the obvious 2026 thing: I wired a vision model into the pipeline. For each archive it renders two screenshots — a reference (the canonical rendering of the source) and a candidate (what we produced) — and asks the model to grade the pair: SAME, MINOR-DIFF, or DEGRADED-CONTENT. CI runs the whole corpus nightly, opens a GitHub issue for every regression, and a green check means we're fine.

It looked rigorous. It had a frozen reference corpus, a pinned model, a CI gate, auto-filed regression issues, a dashboard with burn-down charts. It had the shape of a measurement system. It took me an embarrassingly long debugging session — chasing a 429 here, an unreachable storage endpoint there, a silently swallowed error somewhere else — before I sat back and noticed the actual problem, which had nothing to do with any of those bugs.

Nobody had ever checked whether the judge was right.

The shape of rigor is not rigor

Walk the chain of trust backwards. The gate fires when the model's verdict for an archive disagrees with the expected verdict stored in the corpus. So the gate is only as good as two things: the model's verdict being correct, and the expected verdict being correct.

Where did the expected verdicts come from? I wrote them. Based on what? On my assumption, at corpus-creation time, that those captures were good — an assumption I never verified by looking. Where does the model's verdict come from? A frontier vision model that is extremely confident, extremely fluent, and has been graded by exactly no one. Not once. I had never put its SAME/DEGRADED calls next to a human's and counted the disagreements.

So what was the gate actually measuring? Not "is this archive faithful." It was measuring "did the model's opinion change relative to a label I guessed." Those are different quantities. Only one of them is the thing I care about, and it was not the one being measured. The burn-down chart was real. It was burning down the wrong number.

This is the trap, and it is a 2026-specific trap: a model output is always available, always formatted, always confident, always on time. It has every surface property of an answer. The one property it does not come with — and the one nothing in the pipeline supplied — is a reason to believe it. An automated judge you have never graded is not a measurement. It is an opinion with a progress bar.

The fix was the least clever thing in the building

The instinct, when you discover your judge is uncalibrated, is to build a better judge. Ensemble it. Add a second model to grade the first. Tighten the prompt. Add a pixel-diff floor. More automation, pointed at the same hole.

None of that works, for a reason that is almost insultingly simple: you cannot bootstrap correctness from a system that has never touched ground truth. A second model grading the first is two opinions and still zero facts. At some point a human has to look at the pixels and say "that one's faithful, that one's broken." There is no automation that produces the first true label. Someone has to make it.

So I built the dumbest possible thing. One self-contained HTML file. It scans the screenshot pairs on disk and lays them out: reference on the left, candidate on the right, forty-two of them, full size. Four keys:

1  faithful            candidate matches the control
2  minor               cosmetic differences only
3  degraded            real content missing / broken
4  source-unavailable  the REFERENCE itself is blank/blocked

It autosaves to localStorage so a misclick or a closed tab doesn't cost you the session. It has one button: Export labels, which hands back a JSON file. No server. No build step. No network. No model. A person, a grid, and an afternoon — except it isn't even an afternoon, it's about twenty minutes, because looking at two pictures and pressing a number is fast when you stop pretending it needs to be a pipeline.

That JSON is the entire point. It becomes two things at once. First, it is the authoritative source for the corpus's expected verdicts — the guesses get replaced with observations. Second, and more important, it is the answer key. Now I can run the model over the same set and compute the only metric that was ever worth computing: how often does the judge agree with a human who actually looked? If that number is 96%, the gate is real and I can trust it to run unattended. If it's 70%, the gate was theater and every burn-down chart I drew was a lie I told myself in good faith. Either way I finally know, instead of believing.

The fourth label — source-unavailable — earns its place too. A large fraction of "bad" archives in any real corpus aren't your bug at all; the thing you were comparing against was itself a CAPTCHA wall or a blank error page. An automated judge comparing two blanks will happily call them identical and score it green. A human looking at two blank rectangles says "neither of these is anything" in half a second. That distinction — my system broke this vs. the source was already gone — is invisible to the judge and obvious to the eye, and it is the difference between a defect you must fix and one you must merely record.

Calibrate before you delegate

The specifics here are about screenshots and an archiving pipeline, but the failure generalizes to every place we are currently wiring model judgment into something that runs on its own. Model-graded evals. LLM-as-judge test suites. Agents whose stopping condition is another model saying "looks done." Quality gates whose definition of quality is a model's say-so. They share an architecture, and they share this hole.

The seductive property is the same every time: the system keeps producing confident, well-formatted verdicts forever, with no external signal about whether they're any good, and the more capable the model gets the stronger the pull to skip the check — because the output reads as authoritative, and building the boring answer key feels like a step beneath a system this sophisticated. It is not beneath it. It is the foundation it was supposed to be standing on.

So the rule I'm taking out of this, and the one I'd offer you: before an automated judge is allowed to gate anything, it owes you a number — its agreement rate against human-labeled ground truth, on a sample you actually looked at. Not vibes. Not "the model is really good now." A measured rate, recomputed when the model or the task changes. Calibrate, then delegate. In that order, every time.

Ground truth is cheap. It is one person and twenty minutes and a deliberately stupid HTML file. The reason it almost never gets made is not cost. It's that it doesn't feel like engineering — it's unscalable, unautomatable, and faintly embarrassing to admit you needed it. Make it anyway. It is the only part of the whole apparatus that is actually true, and everything else is just confidence until you do.
